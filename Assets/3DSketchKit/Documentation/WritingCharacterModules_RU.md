# Написание модулей персонажа

Модули персонажа — это типизированные `ScriptableObject`-ассеты. Каждый модуль должен принадлежать ровно одной host-системе.

## Контракт

- Модули локомоушна наследуют `LocomotionModuleAsset`.
- Модули способностей наследуют `AbilityModuleAsset`.
- Боевые модули наследуют `CombatModuleAsset`.
- Модули инвентаря наследуют `InventoryModuleAsset`.
- Модули экипировки наследуют `EquipmentModuleAsset`.
- Модули управления наследуют `ControlModuleAsset`.
- Модули взаимодействия наследуют `InteractableModuleAsset`.

Добавьте `CharacterSystemModuleAttribute`, чтобы окно интеграции показывало понятное имя и группу host-системы.

```csharp
[CharacterSystemModule(CharacterSystemKind.Combat, "Melee Combat")]
public sealed class MeleeCombatModule : CombatModuleAsset
{
}
```

## Правила

- Не пишите один модуль, который владеет сразу несколькими системами.
- Не редактируйте скрытое/серийализуемое состояние другой host-системы напрямую.
- Для межсистемного взаимодействия используйте публичные API host-систем.
- Проверяйте наличие нужных сокетов, профилей и зависимостей.
- Освобождайте временные объекты и подписки на события в `Shutdown`.
- Предпочитайте маленькие узкие модули вместо одного «бог-модуля».

## Руководство для начинающих: как использовать модули без программирования

Большинству авторов достаточно работать ассетами модулей и профилей.

### A) Где физически лежат модули

Модули — это `ScriptableObject` `.asset`, по умолчанию их можно держать так:

```text
Assets/3DSketchKit/Runtime/Modules/
```

Эксперименты пользователя лучше держать отдельно:

```text
Assets/UserContent/Characters/<CharacterName>/Modules/
```

Так проще обновлять ассет и не смешивать «ядро» и пользовательский контент.

### B) Как создать asset модуля (2 способа)

#### Способ 1: меню `Create`

Если модуль объявляет `CreateAssetMenu`, создайте через:

`Create → 3D Sketch Kit → Modules → ...`

#### Способ 2: окно интеграции

Откройте:

`3D Sketch Kit → Characters → Integrate Character Module`

Там показываются найденные конкретные типы модулей, можно создать `.asset` в выбранную папку.

### C) Подключение через профили (предпочтительно)

Профили содержат списки модулей:

- `LocomotionProfile` → локомоушн-модули
- `CombatProfile` → боевые модули
- `InteractionProfile` → интеракции

Типичный процесс:

1. Выберите профиль `.asset` в Project.
2. Добавьте ссылки на модули в соответствующие списки.

### D) Подключение напрямую на host-систему (быстрый эксперимент)

У каждого host-компонента может быть дополнительный список модулей для черновых настроек.

Для финальных пресетов обычно лучше держать всё в профилях, чтобы переиспользовать между prefab.

### E) Проверка из Inspector

На корне prefab с `CharacterEntity` используйте кнопку валидации (если доступна в вашей сборке редактора), чтобы быстро увидеть проблемы по зависимостям и сокетам.

### F) Когда всё-таки нужен код

Если нужно новое поведение:

1. Выберите ровно один тип базового класса (`CombatModuleAsset`, и т.д.).
2. Добавьте `[CharacterSystemModule(CharacterSystemKind..., "Читаемое имя")]`.
3. Основную логику проверки держите в `Validate(...)`, а межсистемные эффекты — через публичные API host-систем.
